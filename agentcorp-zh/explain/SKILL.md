# Explain

Explain 处理的是：**"我没看懂这个。"**

用户可能缺的是整个上下文。这很正常。他可能没读过 repo、报告、issue、前置调查，也不熟悉这块知识体系。你的任务是让他在缺上下文的情况下，仍然能看懂眼前这个东西。

保持简单。把它当作一层翻译：把不透明材料翻译成可用的理解；不要把它变成 research role、review role、教程生成器或交付 phase。

## 哲学

回答 **what**：

- 这是什么？
- 它处在哪个上下文里？
- 这句话、这份报告、这个 finding、这个指标、这条日志或这个决策在说什么？
- 它默认了哪些背景知识？
- 用户应该从里面带走什么理解？
- 哪些仍然未知，哪些只是推断？

不默认展开 detailed **how**。只有当机制是理解 what 的必要条件时，才补一点 how。不要把一个局部没看懂，扩写成完整 walkthrough、repo 导览、实现课或操作步骤，除非用户明确要。

## 上下文

解释前先检查足够的 source material。用户指向报告，就先读报告结构和相关段落；报告依赖 repo，就检查足够的 repo 上下文，弄清它在说哪块东西。用户指向术语、句子、finding、日志、指标或 artifact 摘录，就在可用时读取周边段落、原始 artifact、命令输出或任务记录。

然后选择能让对象变得可理解的最小上下文：

- 一个知识点的上下文，是它所在的知识体系。
- 一份报告的上下文，是 repo 加上报告整体论证。
- 一个 review finding、状态、测试结果、日志、指标或 artifact 摘录的上下文，是任务历史和原始产物。
- 一个术语或句子的上下文，是它所在段落，以及这个段落属于的更大对象。

从用户选中的东西开始，只向外扩展到刚好够理解为止。如果必要上下文不可用，就说明缺什么，并把解释标成推断。

## 边界

当 `probe`、`brainstorm`、`grill`、`explain`、`walkthrough` 的边界不清楚时，读取 `../_shared/thinking-system.md`。

当用户想理解眼前已有的东西是什么意思，留在 `explain`。只有真实任务变成完整走读、发现盲区、塑造选项或压测方案时，才交给其他 skill。

## 默认回答形状

默认用自然语言。通常这个最小形状就够：

1. **直译成人话**：把卡住人的东西改写成直接的话。
2. **上下文**：说明它在知识体系、repo、报告、任务或 artifact 里的位置。
3. **默认背景**：只解释眼下必须知道的术语和前提。
4. **该带走什么**：说明用户应该因此理解什么或判断什么。
5. **边界**：区分已确认事实、推断和未知。

小问题一段话讲完。大报告按 section 重复这个形状。

## 风格

- 默认用户聪明，只是缺上下文。
- 不要居高临下。避免"显然""很简单""只是""明显"。
- 宁可短而准，不要面面俱到。
- 例子只在能让 what 更清楚时使用。
- 除非确实有助于理解，否则不要上 Mermaid、图、长流程或实现细节。
- 不要编造 repo、报告或领域上下文；有 source material 时使用 source material。

## 输出

默认 inline 回答。

只有当用户要求写下来、做成文档、方便以后重读、分 section 解释长报告，或需要沉淀给后续使用时，才写 artifact。落库路径使用 `explain/<topic-slug>.md`；多文件集合使用 `explain/<topic-slug>/`。
